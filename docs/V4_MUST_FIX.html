v4.0.0 우선 수정 요청 (P0)

대상: terraform-provider-samsungcloudplatformv2 v4.0.0 (released, bc2a1a6)

블랙박스 terraform 커버리지로 확인한 필수(P0) 결함 12건입니다. 기준 = "이게 없으면 리소스를 생성할 수 없거나, destroy 시 리소스가 누수되거나, 프로바이더가 멈춤/패닉". 각 항목은 문제 → 수정 전 소스 → 수정 후 소스로 정리했습니다. 코드 위치는 v4.0.0 소스 기준.

A. 생성 자체가 막힘 (리소스 사용 불가)

virtualserver_server — state 미지정 시 create 항상 실패

문제. state(Optional+Computed)를 설정하지 않으면(문서 기본 사용법) apply가 "Invalid server state. Server state must be 'ACTIVE' during creation."로 실패. 컴퓨트 티어 전체가 사실상 생성 불가. (미지정 시 plan에서 unknown 인데 가드가 null 이 아닌 빈 문자열로 읽어 분기됨.)

수정 전 — service/virtualserver/server.go

// :290  State 스키마 — Default / plan modifier 없음
common.ToSnakeCase("State"): schema.StringAttribute{ Optional: true, Computed: true },

// :1614  create 가드 — IsUnknown 미검사
if !plan.State.IsNull() {
    if plan.State.ValueString() != "ACTIVE" {
        resp.Diagnostics.AddError("Error Creating Server", "...State: "+plan.State.ValueString())
        return                       // unknown -> "" -> 항상 실패
    }
}

수정 후

common.ToSnakeCase("State"): schema.StringAttribute{
    Optional: true, Computed: true,
    PlanModifiers: []planmodifier.String{ stringplanmodifier.UseStateForUnknown() },
},

if !plan.State.IsNull() && !plan.State.IsUnknown() {     // unknown은 통과
    if plan.State.ValueString() != "ACTIVE" {
        resp.Diagnostics.AddError("Error Creating Server", "...")
        return
    }
}

수정 후

if !request.Description.IsNull() {           // 설정된 경우에만 첨부
    req.Description = *scpvpc.NewNullableString(request.Description.ValueStringPointer())

B. destroy 실패 → 리소스 누수 (비용/쿼터/보안)

loadbalancer — create가 ACTIVE 대기 안 함 → destroy 실패 → 과금 LB 누수

문제. Create가 LB가 CREATING 인 채로 반환 → 직후 destroy가 "not in a deletable state (state: CREATING)"로 실패 → 과금 LB 누수 + 이름 점유.

수정 전 — service/loadbalancer/loadbalancer.go

// :247 Create — 폴링 없이 바로 반환
r.client.CreateLoadbalancer(ctx, plan)
resp.State.Set(ctx, plan)

// :344 Delete — deletable 대기 없이 바로 삭제
r.client.DeleteLoadbalancer(ctx, id)

수정 후

r.client.CreateLoadbalancer(ctx, plan)
client.WaitForStatus(ctx, r.client, []string{"CREATING"}, []string{"ACTIVE"}, getFunc) // ERROR 단락
resp.State.Set(ctx, plan)

// Delete: 전이상태면 deletable까지 대기 후 삭제, 404는 성공
client.WaitForStatus(ctx, r.client, []string{"CREATING","EDITING"}, []string{"ACTIVE"}, getFunc)
r.client.DeleteLoadbalancer(ctx, id)
// 주의: v4 WaitForStatus 시그니처에 timeout/delay/minTimeout/maxConsecutiveErrors 추가됨

C. 자동화 멈춤 / 프로바이더 패닉

VPC waiter — 빈 Pending + 120m → ERROR/parked 상태에서 2시간 행

문제. VPC 계열 status waiter가 모두 Pending=[] 로 호출 + WaitForStatus 120분 → transit_gateway/private_nat 등이 ERROR/parked에 빠지면 ERROR 단락 없이 2시간 폴링 → apply/destroy 멈춤, 자동화에선 teardown까지 막혀 누수 위험. (transit_gateway / transitgateway_vpcconnection / vpc_transit_gateway_rule / transit_gateway_firewall_connection / private_nat 전체 동일.)

수정 전 — client/common.go + 각 waiter 호출부

waitForTgwtStatus(ctx, client, id, []string{} /* pending 비어있음 */, []string{"ACTIVE"})
// refresh 에 ERROR 단락 없음, Timeout = 120m -> 어떤 상태든 2시간 폴링

수정 후

waitForTgwtStatus(ctx, client, id, []string{"CREATING","EDITING"}, []string{"ACTIVE"})
// refresh: if state == "ERROR" { return fatal }      // 즉시 단락
// 네트워크 리소스 Timeout 120m -> 15~20m

에러 응답 파싱 — 500/특정 응답에서 프로바이더 패닉 (unchecked type assertion)

문제. 에러 디테일 파싱의 검사 없는 타입 단언 → 500 응답의 detail 원소가 string이 아니면 프로바이더 패닉(프로세스 크래시). v4 전체 스윕에서 일부 샤드가 이 패닉으로 산출물 없이 크래시함.

수정 전 — client/common.go:142 (GetDetailFromError)

for _, d := range detail.([]interface{}) {
    details = append(details, d.(string))   // 비-string 원소면 panic
}

수정 후

for _, d := range detail.([]interface{}) {
    if s, ok := d.(string); ok { details = append(details, s) }   // comma-ok, 비-string 스킵
}
// 동류: common/virtualserver/virtualserver.go 의 err.(*scpsdk.GenericOpenAPIError) 도 comma-ok로
