# SCP Terraform Provider — v4.0.1 기준 **반드시 고쳐야 할 결함** 리포트

> **검증 기준:** upstream **v4.0.1** provider 소스 직접 대조 (`terraform-provider-samsungcloudplatformv2` tag `v4.0.1`).
> **작성 맥락:** SCP API/Terraform regression 플랫폼이 추적 중인 45개 provider 결함(`v4-still-lacks`)을
> v4.0.1과 1:1 reconcile 한 결과입니다. v4.0.1이 이미 해결한 5건은 맨 아래 §A 에 정리(우리 tracker에서 close 완료),
> 아래 §1–§4 는 **v4.0.1 에도 그대로 남아 있어 반드시 고쳐야 하는** 결함을 심각도순으로 정리한 것입니다.
> 각 항목에 **v4.0.1 의 `file:line` 근거**를 달았으니 그대로 확인/수정 가능합니다.

## §1. P0 — Critical (운영 차단: destroy 불가 / panic / drift / data-loss)

### #60 — `vpc_cidr` Delete 미구현 **[확인됨]**
- **v4.0.1:** `service/vpc/vpc_cidr.go:252` → `Delete()` 본문이 `// TODO: Implement Delete function` + `resp.Diagnostics.AddError("Delete Not Implemented", …)`.
- **영향:** `terraform destroy` 불가 → CIDR 리소스 leak/orphan, state 꼬임.
- **수정:** Delete 실제 구현 (해당 CIDR 해제 API 호출 + state 제거).

### #63 — `transit_gateway_firewall` / `_firewall_connection` Update no-op **[확인됨]**
- **v4.0.1:** `service/vpc/transit_gateway_firewall.go:291` → `Update()` 가 `resp.Diagnostics.AddWarning("Update Not Implemented", …)` 만 하고 반환.
- **영향:** `apply` 가 성공처럼 보이지만 실제 변경 미반영 = **silent config drift** (가장 위험한 류).
- **수정:** Update 구현, 또는 변경 속성에 `RequiresReplace`.

### #76 — VPC waiter 가 빈 `Pending` 전달 → 최대 120분 hang **[확인됨]**
- **v4.0.1:** `service/vpc/transit_gateway.go:196,319` → `waitForTgwtStatus(ctx, …, []string{}, []string{"ACTIVE"})` / `…, []string{}, []string{"DELETED"})`. pending 집합이 `[]string{}`.
- **영향:** 리소스가 `ERROR` 등 비정상 상태로 가도 pending=∅ 이므로 “예상치 못한 상태”로 조기 실패하지 못하고 target 만 기다리다 **timeout(120m) hang**. TGW·firewall·rule·private_nat 공통.
- **수정:** 실제 진행 상태(`CREATING/DELETING/…`)를 pending 으로 전달하고, pending/target 어디에도 없는 상태는 즉시 error.

### #77 — `loadbalancer` Delete 가 완전 삭제를 대기하지 않음 **[확인됨]**
- **v4.0.1:** `service/loadbalancer/loadbalancer.go:363` `Delete()` 에 wait-until-gone 없음. (Create-wait-`ACTIVE` 는 v4.0.1 에서 추가됨 — Delete 만 누락.)
- **영향:** LB 가 아직 삭제 중인데 Terraform 은 완료로 간주 → 의존 subnet/VPC destroy 시 **409**.
- **수정:** Delete 후 리소스가 사라질 때까지 polling.

### #79 / #93 — `dns_private_dns` Delete 가 VPC unbind/wait 없이 삭제 **[확인됨]**
- **v4.0.1:** `service/dns/private_dns.go:395` `Delete()` 가 unbind/disconnect 및 wait 없이 바로 `DeletePrivateDns`.
- **영향:** hosted zone 이 VPC 에 연결된 상태로 남아 **409**, 자식/의존 리소스 teardown 차단.
- **수정:** VPC 연결 해제 → disconnect 확인(wait) → delete.

### #88 — `filestorage_replication` Delete 가 정책 pause 없이 삭제 **[확인됨]**
- **v4.0.1:** `service/filestorage/replication.go:342` `Delete()` 가 `PauseVolumeReplication` 선행 없이 바로 `DeleteVolumeReplication`.
- **영향:** 활성 복제 정책 삭제 실패/오류.
- **수정:** pause policy → delete → wait-gone 순서.

### #58 — `iam_access_key` 가 enabled 상태에서 destroy 불가 **[확인됨]**
- **v4.0.1:** `service/iam/accesskey.go:386` `Delete()` 가 disable 선행 없이 삭제.
- **영향:** 활성 access key 를 `terraform destroy` 로 제거 불가.
- **수정:** disable → delete.

### #26 — `GetStatusCodeFromError` unchecked type assertion → panic **[확인됨]**
- **v4.0.1:** `common/virtualserver/virtualserver.go:263` → `body := err.(*scpsdk.GenericOpenAPIError).Body()` (가드 없음), `:267` `errors.([]interface{})`, `:268` `err.(map[string]interface{})` 모두 unchecked.
- **영향:** 예상치 못한 에러 타입/구조에서 provider **panic** (crash).
- **수정:** comma-ok 가드. ※ 같은 파일군 `client/common.go:112` `GetDetailFromError` 는 v4.0.1 에서 이미 `if genericErr, ok := err.(*…); ok` 로 가드됨 — **동일 패턴을 `GetStatusCodeFromError` 에도 적용**하면 됨 (#80 backup 디코드 crash 도 이 경로).

### #54 / #62 — 모델이 raw Go 타입 → null/unknown 에서 Value Conversion **[확인됨]**
- **v4.0.1:** 예) `client/cloudmonitoring/model.go:130` → `NotificationRecipient []NotificationRecipient`(raw slice). 22개 필드군 동일 패턴.
- **영향:** API 가 null 반환하거나 plan-time unknown 일 때 *“Received null/unknown value, target type cannot handle”* → **create/import 실패(crash)**.
- **수정:** `types.List`/`types.Object` 로 전환. ※ v4.0.1 이 DBaaS `instance_groups` 는 신규 util `common/database/instance_group.go`(`MapInstanceGroupsList`)로 해결했음 — **같은 접근을 cloudmonitoring 등 잔여 필드에도** 확장 권장.

### (data-loss 후보) #41 — `FileStorage` access_rules 가 외부 추가 노드를 삭제 **[재확인 권장]**
- **위치:** `service/filestorage/volume.go`. 외부(예: K8s autoscaler)가 추가한 access rule 을 reconcile 시 제거 → 운영 트래픽 영향.
- **수정:** 외부 추가 분 보존(merge) 또는 명시적 ignore.

## §2. P1 — Import / State 정확성

### #52 — Read 가 404 에서 state 제거 안 함 (75/83 리소스) **[확인됨(샘플)]**
- **v4.0.1:** 예) `service/vpc/publicip.go:202` `Read()` 가 GetXxx 실패 시 `RemoveResource` 없이 error 반환.
- **영향:** 리소스가 콘솔/외부에서 삭제되면 state 에 stale 하게 남아 **영구 plan 오류** (recreate 불가).
- **수정:** 404/NotFound → `resp.State.RemoveResource(ctx)` 후 return (전 리소스 표준화).

### #62 (잔여) — raw-type import 실패 지속 **[확인됨]**
- §1 #62 와 동일 근원. DBaaS 외 raw-type 필드 보유 리소스는 `terraform import` 시 Value Conversion 으로 실패.
